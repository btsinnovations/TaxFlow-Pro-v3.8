# TaxFlow Pro v3.10 Build-Design Research

## Goal for v3.10

Transform TaxFlow Pro from a developer-run Python project into a **single-user desktop installer** that runs offline with one command.

## Current state (v3.9.2)

- Backend: FastAPI + SQLAlchemy + SQLite/SQLCipher + Python 3.10+
- Frontend: Vite + React, served via `npm run dev`
- Launch: `start.sh` (Unix) or manual `uvicorn backend.api:app` + `npm run dev`
- Dependencies: Tesseract, Poppler, Python packages, Node packages
- Local assets: `.local_secret`, `taxflow.db`, `backups/`, `uploads/`, `ml/`

## Candidate packaging strategies

### 1. PyInstaller (recommended for v3.10)

**Pros:**
- Mature, well-documented.
- Bundles Python interpreter + site-packages into one executable.
- Works on Windows, macOS, Linux.
- Can embed data files and launch the FastAPI server from a `--onefile` or `--onedir` build.

**Cons:**
- Large binary (200–400 MB for ML + scientific Python stack).
- Startup time can be slower for `--onefile`.
- Anti-virus false positives possible on Windows.
- Need to handle frontend build as static files (build Vite, serve from FastAPI).

**Fit for TaxFlow:**
- Best because the app is Python-centric. The frontend can be built once and served as static assets.

### 2. cx_Freeze

**Pros:**
- Python-native, cross-platform.
- MSI/installer support on Windows via bdist_msi.
- Smaller than PyInstaller in some cases.

**Cons:**
- Less community tooling than PyInstaller.
- Can struggle with compiled extensions (pydantic, SQLCipher, bcrypt, cryptography).

### 3. Briefcase (BeeWare)

**Pros:**
- Produces native installers per OS.
- Clean project structure.

**Cons:**
- Requires app to conform to Briefcase patterns.
- Smaller ecosystem; may not handle FastAPI/Vite smoothly.

### 4. PEX / shiv

**Pros:**
- Single-file Python executable.
- Good for CLI tools.

**Cons:**
- Does not bundle Python interpreter.
- Not ideal for GUI/frontend apps on end-user machines.

### 5. Electron + embedded Python (Tauri alternative)

**Pros:**
- True native desktop shell.
- System tray, auto-start, native menus.

**Cons:**
- Heavy (Electron bundle ~100 MB on top of Python).
- More complex IPC between frontend and Python backend.
- Overkill for v3.10; better for Phase 5.

## Recommended v3.10 architecture

### Short-term: PyInstaller + Vite static build

1. **Build frontend:**
   ```bash
   cd frontend
   npm run build
   ```
   Output: `frontend/dist/` with static HTML/JS/CSS.

2. **Serve frontend from FastAPI:**
   Add `StaticFiles` mount in `backend/api.py`:
   ```python
   from fastapi.staticfiles import StaticFiles
   app.mount("/", StaticFiles(directory="frontend/dist", html=True), name="static")
   ```
   Ensure API routes still work under `/api/*`.

3. **PyInstaller spec:**
   - Bundle backend code, alembic, frontend dist, tesseract binaries, poppler binaries.
   - Entry point: a small Python launcher that:
     - Sets `TAXFLOW_LOCAL_ROOT` to a user-writable directory (e.g., `%APPDATA%/TaxFlow` on Windows, `~/.taxflow` on Unix).
     - Runs Alembic migrations.
     - Starts Uvicorn on `127.0.0.1:8000`.
     - Opens the default browser to `http://localhost:8000`.

4. **Installer wrappers:**
   - Windows: Inno Setup or NSIS around the PyInstaller directory.
   - macOS: `.app` bundle + `create-dmg`.
   - Linux: AppImage or tarball.

### Data directory strategy

Critical: the executable directory is read-only after installation. All user data must live in a separate writable location:

- Windows: `%LOCALAPPDATA%/TaxFlowPro`
- macOS: `~/Library/Application Support/TaxFlowPro`
- Linux: `~/.local/share/TaxFlowPro`

Subdirectories:
- `db/` — SQLite/SQLCipher database
- `backups/` — encrypted snapshots
- `uploads/` — imported statements
- `ml/` — trained models
- `.local_secret` — JWT signing key
- `logs/` — runtime logs

### External binary bundling

- **Tesseract OCR:** vendor `tesseract.exe` / `tesseract` + tessdata.
- **Poppler (pdftotext / pdfimages):** vendor binaries.
- **SQLCipher:** already bundled via `sqlcipher3-wheels`; verify it loads from PyInstaller.

### Auto-launch browser

Use Python's `webbrowser.open("http://127.0.0.1:8000")` after server is healthy. Provide a small tray/status window so the user knows the server is running.

### First-run wizard

- Create master password.
- Optional SQLCipher encryption.
- Self-test (Tesseract, Poppler, DB writable).
- Already partially implemented in `backend/local/bootstrap.py`; wire it to a frontend onboarding flow.

## Open questions for Josh

1. **Single OS first?** Windows-only installer for v3.10, or parallel Windows/macOS/Linux?
2. **Native shell or browser?** Browser auto-launch is simpler; Electron/Tauri shell is heavier but more polished.
3. **Include Tesseract/Poppler?** Increases installer size by ~100 MB each. Alternative: detect system install and prompt user.
4. **Auto-update?** Deferred to Phase 5, but should the v3.10 build leave hooks?
5. **Code signing?** Windows SmartScreen requires EV code signing; macOS notarization required. Both cost money and time.
