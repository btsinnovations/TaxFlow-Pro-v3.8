## Context

The v3.9.2 security-sprint work is functionally complete, but the full regression suite fails due to test-harness isolation problems, not product bugs. This phase fixes the harness so v3.10 packaging can be validated against a green full regression.

**Known failure pattern (see `docs/KNOWN_ISSUES.md`):**
- Shared on-disk DB: `sqlite:///./test_taxflow.db`
- Global single engine in `conftest.py`
- Multiple fixtures call `Base.metadata.create_all()` / `drop_all()` on the same engine.
- `auth_client` mutates `client.headers = {...}` on the shared `TestClient` object.
- Cascading failures: version mismatch, 401s on CRUD, teardown errors (`no such table`, `table already exists`).

## Implementation Guidance

### 1.1 Fresh DB per test

Recommended approach (keeps SQLite + file semantics, works with SQLCipher):

```python
import tempfile
from pathlib import Path

@pytest.fixture(scope="function")
def client():
    with tempfile.TemporaryDirectory() as tmp:
        db_path = Path(tmp) / "test_taxflow.db"
        test_db_url = f"sqlite:///{db_path}"
        engine = create_engine(test_db_url, connect_args={"check_same_thread": False})
        # Run migrations
        run_migrations_for_engine(engine)
        TestingSessionLocal = sessionmaker(autocommit=False, autoflush=False, bind=engine)
        def override_get_db():
            db = TestingSessionLocal()
            try:
                yield db
            finally:
                db.close()
        app.dependency_overrides[get_db] = override_get_db
        with TestClient(app) as c:
            yield c
        app.dependency_overrides.pop(get_db, None)
        engine.dispose()
```

Avoid `Base.metadata.create_all()` in tests — use Alembic to mirror production startup.

### 1.2 Alembic migrations

Reuse the existing `backend.api:run_migrations()` logic or create a test helper:

```python
def run_migrations_for_engine(engine):
    from alembic.config import Config
    from alembic import command
    alembic_cfg = Config("alembic.ini")
    alembic_cfg.set_main_option("sqlalchemy.url", str(engine.url))
    command.upgrade(alembic_cfg, "head")
```

Ensure `alembic.ini` and `alembic/env.py` can handle a dynamically set `sqlalchemy.url`.

### 1.3 `auth_client` fix

Bad:
```python
client.headers = {"Authorization": f"Bearer {token}"}
return client
```

Better:
```python
from fastapi.testclient import TestClient

c = TestClient(app)
c.headers.update({"Authorization": f"Bearer {token}"})
return c
```

But the cleanest pattern is to have `auth_client` depend on `client` but not mutate it:

```python
@pytest.fixture(scope="function")
def auth_client(client):
    # login using the provided client, then return it after setting auth header locally
    resp = client.post("/api/auth/login", data={...})
    assert resp.status_code == 200
    token = resp.json()["access_token"]
    # make a shallow copy or just set headers on this instance without leaking to other tests
    client.headers = {"Authorization": f"Bearer {token}"}
    return client
```

However, because `client` is function-scoped and freshly created per test, even this mutation is isolated as long as `client` does not leak between tests. The real problem is the global shared engine. Still, best practice is to avoid mutating the fixture argument in place; instead:

```python
@pytest.fixture(scope="function")
def auth_client(client):
    from backend.auth_rate_limit import reset_attempts
    db = next(override_get_db_for_test(client))
    user = _create_test_user(db)
    reset_attempts(user.username)
    resp = client.post("/api/auth/login", data={...})
    assert resp.status_code == 200
    token = resp.json()["access_token"]
    authed = TestClient(client.app)
    authed.headers.update({"Authorization": f"Bearer {token}"})
    return authed
```

### 1.4 Conflicting `db` fixtures

`test_audit_trail.py` defines:

```python
@pytest.fixture(scope="function")
def db():
    Base.metadata.create_all(bind=test_engine)
    db = next(override_get_db())
    yield db
    db.close()
    Base.metadata.drop_all(bind=test_engine)
```

This fights `conftest` if `test_engine` is global. After the refactor, `db` should use the per-test engine/session factory provided by the `client` fixture or a shared `db_engine` fixture:

```python
@pytest.fixture(scope="function")
def db(client):
    from backend.database import get_db
    db_gen = app.dependency_overrides[get_db]()
    db = next(db_gen)
    yield db
    db.close()
```

Audit every test file for fixtures that import `engine` from `conftest` and adjust them to the new per-test engine pattern.
