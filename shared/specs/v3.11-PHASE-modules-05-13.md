## 3.11.05 — Check Register

### Scope
Specialized register view and API for checking accounts: check number tracking, payee/memo, status (outstanding/cleared/void), running balance, and printable check layout.

### Backend
- `backend/accounting/checks.py` — domain logic for check number allocation, voiding, status transitions.
- `backend/routers/checks.py` — FastAPI routes:
  - `GET /api/accounts/{id}/checks` — list checks
  - `POST /api/accounts/{id}/checks` — issue new check (auto-assign next number)
  - `PATCH /api/checks/{id}` — update status (cleared/void)
  - `POST /api/checks/{id}/void` — void a check
- Add `checks` table: `id`, `account_id`, `tenant_id`, `check_number`, `payee`, `memo`, `amount`, `date`, `status`, `void_reason`, `created_at`.

### Frontend
- `frontend/src/components/register/CheckRegister.tsx` — register grid with check number, payee, status, amount.
- `frontend/src/components/checks/PrintCheck.tsx` — printable check layout (8.5x3.5in standard check).

### Tests
- `backend/tests/test_checks.py` — issue, void, duplicate-number guard, running balance.

### Acceptance Criteria
- [ ] New checking transactions can be tagged with a check number.
- [ ] Voiding a check reverses its effect on account balance.
- [ ] Duplicate check number in same account rejected.
- [ ] Running balance reflects only cleared + outstanding checks.

## 3.11.06 — Loans / Credit Lines / Investments

### Scope
Track liability and asset accounts beyond simple depository accounts: loans, credit cards, lines of credit, and investment holdings.

### Backend
- `backend/accounting/liabilities.py` — loan amortization, interest accrual, minimum payment calculation, credit-line available balance.
- `backend/accounting/investments.py` — lot tracking, realized/unrealized gain calculation.
- `backend/routers/liabilities.py` — CRUD + amortization schedule endpoints.
- `backend/routers/investments.py` — lot and holding endpoints.

### Data Model
- `loan_schedules`: `id`, `account_id`, `original_principal`, `rate`, `term_months`, `start_date`, `payment_amount`, `schedule_json`.
- `investment_lots`: `id`, `account_id`, `symbol`, `shares`, `cost_basis`, `acquisition_date`, `sale_date`, `sale_proceeds`.

### Frontend
- Account detail tabs on `frontend/src/components/accounts/AccountDetail.tsx`.
- `frontend/src/components/liabilities/LoanSchedule.tsx`.
- `frontend/src/components/investments/Holdings.tsx`.

### Tests
- `backend/tests/test_liabilities.py` — amortization math, payment allocation.
- `backend/tests/test_investments.py` — FIFO cost basis, gain/loss.

### Acceptance Criteria
- [ ] Loan amortization schedule matches standard formula.
- [ ] Credit-line available balance = limit - current balance.
- [ ] Investment lots support FIFO cost-basis.
- [ ] Gain/loss calculated correctly on sold lots.

## 3.11.09 — Bank Reconciliation

### Scope
Match imported bank statement transactions against ledger transactions and mark cleared items.

### Backend
- `backend/accounting/reconciliation.py` — import parsing (OFX, QFX, CSV), matching algorithm, discrepancy tracking.
- `backend/routers/reconciliation.py` — upload statement, run match, confirm/unconfirm matches.

### Data Model
- `reconciliation_imports`: `id`, `account_id`, `import_date`, `statement_date`, `statement_balance`, `filename`.
- `reconciliation_matches`: `id`, `import_id`, `ledger_tx_id`, `statement_tx_id`, `match_type` (auto/manual/unmatched), `status`.

### Frontend
- `frontend/src/components/reconciliation/Reconciliation.tsx` — side-by-side statement vs ledger.

### Tests
- `backend/tests/test_reconciliation.py` — exact match, fuzzy match, unmatched detection, balance reconciliation.

### Acceptance Criteria
- [ ] OFX/QFX/CSV statement import works.
- [ ] Auto-matching by amount + date within configurable window.
- [ ] Manual match/unmatch supported.
- [ ] Reconciliation report: cleared balance + outstanding = statement balance.
