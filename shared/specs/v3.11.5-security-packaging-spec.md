**Status:** Draft (awaiting v3.11 merge and branch cut)

**Source Branch:** `v3.11` after Josh merge

**Goal:** Consolidate all deferred hardening and packaging work (v3.9.2 security sprint, v3.10 desktop packaging, v3.11 multi-tenant/RLS) into a single validation + packaging release.

---

## 1. Core Principles

1. **Trust through verification.** Every v3.9.2 security control is re-audited and exercised with focused tests before release.
2. **Packaging is a feature.** The app must install, launch, update, and uninstall like a normal desktop product — no terminal required.
3. **Public trust signals are deferred by default.** Josh’s directive stands: functionality over form for friends/family. Code-signing is staged but not required for private distribution.
4. **User data never lives in the install directory.** All platforms use OS-appropriate local data paths.
5. **Offline-first.** Packaged builds work without cloud authentication or network calls in default mode.
6. **RLS is Phase 2.** Row-level security is designed and partially implemented in v3.11; v3.11.5 hardens and validates it.

## 3. Security Hardening Track (3.11.5.SEC)

### 3.1 Re-validate v3.9.2 Controls

Every control below must pass a focused test suite and a `glm-5.1` validator review.

| ID | Control | Validation Target |
|----|---------|---------------------|
| SEC.01 | Brute-force login protection | Progressive delay + hard lockout on `/api/auth/login` and `/api/auth/login-json` |
| SEC.02 | OS keyring/keychain for `.local_secret` | Secret stored in OS credential store; fallback file never contains plaintext |
| SEC.03 | Encrypted backups by default | Backup encrypted before write; restore decrypts with local secret |
| SEC.04 | PDF upload validation | Size limits, magic-byte/PDF header, MIME check, extension whitelist |
| SEC.05 | Parser subprocess sandbox | Isolated subprocess with CPU/time limits and restricted filesystem |
| SEC.06 | Dependency vulnerability scanning | `pip-audit` / `safety` in CI; fail on high/critical |
| SEC.07 | Secret scanning in CI/pre-commit | Detect committed secrets before push |
| SEC.08 | Audit log tamper detection | Hash-chain + verify endpoint |
| SEC.09 | Append-only audit + DB triggers | Prevent UPDATE/DELETE on `AuditEntry` at DB layer |
| SEC.10 | Refresh token rotation | Rotate refresh on use; revoke family on theft |
| SEC.11 | Security headers + CORS hardening | HSTS, CSP, X-Frame-Options, CORS allowlist, cookie flags |
| SEC.12 | Global rate limiting | Per-IP + per-user limits; 429 with Retry-After |
| SEC.13 | Password breach bloom filter | HIBP-style filter on setup/change |
| SEC.14 | Asymmetric signed audit entries | Ed25519 signing of audit hashes |
| SEC.15 | Request size limits + strict validation | Max body size, field length caps, stricter Pydantic |
| SEC.16 | SAST + SBOM pipeline | `bandit` + CycloneDX generation |
| SEC.17 | Path traversal protection | Sanitize user-derived filenames in uploads/exports |
| SEC.18 | Timing-attack-safe authentication | Constant-time lookup and verification |
| SEC.19 | Temporary file cleanup | Guarantee deletion of uploaded PDFs, OCR images, export temps |
| SEC.20 | Sensitive data in process arguments | Load `DATABASE_URL` password via file or keyring, not argv/env |
| SEC.21 | Dependency confusion mitigation | Pin internal package names; verify hashes |
| SEC.22 | Weak entropy audit | Replace `random` with `secrets` for tokens/keys |
| SEC.23 | YAML safe loading | `safe_load` / `CSafeLoader` everywhere |

### 3.2 New / Remaining Security Gaps

| ID | Task | Description |
|----|------|-------------|
| SEC.24 | Remove `/api/tests/` router from production builds | Test-only routes must not ship in packaged installers |
| SEC.25 | Production build flag | Introduce `TAXFLOW_ENV=production` / `production_mode` flag that disables debug endpoints and test routers |
| SEC.26 | Packaged build secret handling | Ensure packaged app does not ship `.env`, `.local_secret`, or test fixtures |
| SEC.27 | Installer permission hardening | Windows installer does not require admin unless machine-wide install is selected; Linux `.deb` uses correct file modes |
| SEC.28 | Auto-updater security | If updater added, verify signature checks on update payloads (baseline: manual update only) |

### 3.3 Deferred Out of v3.11.5

- MFA/2FA
- Antivirus hook integration
- TPM/HSM key storage

## 5. Multi-Tenant / RLS Track (3.11.5.RLS)

From v3.11 deferred work and v3.9/v3.10 hardening plans.

| ID | Task | Description |
|----|------|-------------|
| RLS.01 | Tenant isolation review | Audit all queries for `tenant_id` / `profile_id` filtering |
| RLS.02 | RLS policies on base tables | Add PostgreSQL RLS policies for `clients`, `accounts`, `transactions`, `documents`, `audit_entries` |
| RLS.03 | Service-role bypass | Ensure migrations/admin operations use a service role that bypasses RLS |
| RLS.04 | Tenant-bound fixtures | Test fixtures create data scoped to a tenant; cross-tenant reads rejected |
| RLS.05 | SQLite fallback behavior | RLS is PostgreSQL-only; SQLite dev builds use application-level tenant scoping |
| RLS.06 | Client-level tenant boundary | Confirm or implement client-as-tenant boundary decision from v3.11 |

### 5.1 Design Decisions Pending Confirmation

These were flagged during v3.11 planning and must be confirmed before RLS implementation:

1. **Tenant boundary:** Client-as-tenant (each client is a isolated tenant) vs profile-as-tenant.
2. **`tenant_id` non-nullable:** Enforce at application and DB layers.
3. **RLS deferral:** Phase 2 if v3.11.5 timeline is tight.
4. **Root `.env` for `DATABASE_URL`:** Keep production connection string in project root `.env`.
5. **SQLite dev fallback:** Keep SQLite for development; PostgreSQL for production RLS.

## 7. Exit Criteria

- Full test suite passes: `pytest backend/tests tests` with 0 failures.
- All security focused suites pass (backup/recovery, audit trail, auth, rate limiting, path traversal, etc.).
- Packaged Windows installer launches, creates local user data dir, and serves UI/API without 401s.
- Packaged Linux `.deb` launches, creates local user data dir, and serves UI/API without 401s.
- macOS build passes smoke test if attempted.
- Production mode disables `/api/tests/` and any debug routers.
- No secrets (`.env`, `.local_secret`, test fixtures) ship in installer artifacts.
- RLS policies pass cross-tenant isolation tests on PostgreSQL, OR an explicit decision defers RLS to v3.12.
- `CHANGES.md` and version files updated to `3.11.5`.
- Validator review (`glm-5.1` or equivalent) signed off.
