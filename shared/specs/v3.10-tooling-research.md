# TaxFlow Pro v3.10 — Third-Party Tooling Integration Research

**Status:** Queued — waiting for weekly usage limit reset
**Scope:** Re-evaluate TaxFlow Pro’s hand-built components and replace them with mature local libraries where appropriate
**Constraint:** Project remains local-first and private. No cloud integration.

---

## 1. Background

TaxFlow Pro v3.7 → v3.9 was built primarily from scratch: custom ML pipeline, custom export generation, custom password hashing, custom session handling, etc. The goal of v3.10 is to selectively replace those hand-built components with well-vetted, offline-capable libraries to reduce code volume, improve validator pass rates, and harden security.

This is analogous to how v3.9 was created: take the best of v3.7 and v3.8, reconcile, and produce a cleaner baseline. v3.10 will do the same at the tooling layer.

---

## 2. Research Questions

For each candidate tool below, Jane must answer:
1. What hand-built component does it replace?
2. What is the exact migration path (files to change, data to migrate, tests to add)?
3. What are the offline/local-first constraints (does it phone home, require network, or pull cloud dependencies)?
4. What is the security impact (better, worse, neutral)?
5. What is the validator impact (fewer custom code paths, new dependency risks)?
6. What is the token/effort cost to integrate?
7. Does it require changes to packaging/installer?

## 3. Candidate Tools

### 3.1 Analytics / Reporting → DuckDB
- Replaces: `backend/routers/export.py`, manual pandas/openpyxl loops, aggregation logic
- Must verify: no telemetry, pure local execution, can read SQLite directly or via SQLAlchemy
- Deliverable: Prototype query that replaces one export format end-to-end

### 3.2 ML Classifier → LightGBM
- Replaces: scikit-learn LogisticRegression in `phase3_pipeline/ml_categorizer.py` and `train_categorizer.py`
- Must verify: trains offline, smaller/faster model, supports imbalanced categories
- Deliverable: Training script comparing accuracy and model size vs. current scikit-learn approach

### 3.3 Model Inference / Serialization → ONNX Runtime
- Replaces: `joblib` serialization and `joblib.load()` in `phase3_pipeline/ml_categorizer.py`
- Must verify: removes pickle/ACE risk, supports scikit-learn export via skl2onnx, inference-only runtime
- Deliverable: ONNX export + inference replacement with fallback behavior preserved

### 3.4 Password Hashing → passlib / argon2-cffi
- Replaces: custom SHA-3_256 + salt in `backend/local/auth.py`
- Must verify: uses Argon2id or bcrypt, no network dependency
- Deliverable: Migration path for existing SHA-3 hashed passwords (backward compatibility or forced re-hash)

### 3.5 Signed Session Tokens → itsdangerous
- Replaces: opaque `secrets.token_urlsafe()` tokens with no validation chain in `backend/routers/auth.py`
- Must verify: timestamped, signed, tamper-evident local tokens
- Deliverable: New token format with expiration and invalidation support

### 3.6 Configuration Management → pydantic-settings
- Replaces: manual `os.environ.get()` and `python-dotenv` scattered across `backend/database.py`, `backend/local/settings.py`, etc.
- Must verify: supports `.env`, validation, type coercion
- Deliverable: Unified settings module

### 3.7 Property-Based Testing → hypothesis
- Replaces: hand-written edge-case tests for date parsing, amount formatting, reconciliation
- Must verify: integrates with pytest, runs offline
- Deliverable: Sample hypothesis test catching a real edge case

### 3.8 PDF Sanitization / Sandboxing → pikepdf
- Replaces: manual parser hardening in `backend/parsers/generic_pdf.py`
- Must verify: can strip JS, embedded files, malformed structures before extraction
- Deliverable: Pre-processing step that rejects dangerous PDF features

### 3.9 Bank Statement Import Parsers
- **OFX → `ofxparse`**
  - Replaces: any hand-rolled OFX parsing in current or planned imports
  - Must verify: no network calls, pure local XML/SGML parsing, handles bank/credit-card statements
  - Deliverable: End-to-end OFX import test with one sample statement
- **QIF → `qifparse` (or custom grammar)**
  - Replaces: hand-rolled QIF parsing
  - Must verify: no network, can parse common QIF record types (Bank, Cash, CCard, Invst)
  - Deliverable: QIF import test or decision that `qifparse` is too unmaintained and a small grammar parser is safer
- **CSV → `pandas` / built-in `csv`**
  - Replaces: any manual CSV splitting
  - Must verify: already offline; add schema-driven validation, formula-injection guard, and date/amount normalization
  - Deliverable: Hardened CSV import with warnings for `=`, `+`, `-`, `@` formula injection prefixes
- **Security angle:** All three formats must route through the existing parser subprocess sandbox (`backend/parsers/sandbox.py`) before reaching the main process.

### 3.19 Local Network Attack Surface
- Origin validation, token binding, local-only secret header between frontend and backend to block malicious browser extensions or other localhost services
- Files: `backend/api.py`, CORS/middleware, frontend API client

### 3.20 Memory-Dump Protection
- Minimize plaintext transaction data retention in Python process memory; consider encrypted in-memory structures for high-sensitivity mode
- Files: `backend/local/crypto.py`, transaction processing paths

### 3.21 Installer / Update Channel Integrity
- Signed manifests, rollback protection, verified hashes for any future auto-updater or downloadable components
- Files: packaging scripts, release pipeline

### 3.22 File-Watcher / Import Directory Security
- Watched folders must not follow symlinks, ignore executables, validate file contents before parsing, and reject oversized/unexpected file types
- Files: future watcher module, `backend/routers/upload.py`

### 3.23 Privilege Separation
- Run PDF/OCR/ML worker under lower-privilege process or user so a parser/ML compromise does not gain full user rights
- Files: `backend/local/guards.py`, subprocess helpers

### 3.24 Clipboard and Drag-and-Drop Sanitization
- Validate dropped files and pasted content before processing; reject unexpected MIME types or paths
- Files: frontend drop zones, any clipboard handlers

### 3.26 Clickjacking / UI Embedding
- Add `X-Frame-Options` / CSP frame-ancestors to prevent the localhost app from being embedded in a malicious iframe
- Already partially covered by TASK-027 but worth validating in v3.10

### 3.27 Error Verbosity / Information Disclosure
- Ensure production mode never returns stack traces or internal paths in API error responses

### 3.28 Race Conditions on Concurrent Imports / Backups
- File-level locking or atomic operations to prevent two imports or backup jobs from corrupting state

### 3.29 API Rate Limiting Beyond Login
- Per-user throttling on upload, export, ML training endpoints

### 3.30 OS Search Indexing of Sensitive Files
- Warn users about Windows Search / Spotlight indexing `taxflow.db` or export directories

### 3.31 Cloud Sync Folder Accidents
- Detect if data directory is inside Dropbox / OneDrive / iCloud and warn user

### 3.32 Printer / Print-to-PDF Exfiltration
- Printed reports may hit cloud printers or temp spool files; add caution or local-only print defaults

### 3.33 File Metadata Leaks
- Strip modification times, PDF metadata, or EXIF from exported reports where applicable

### 3.34 Unencrypted Swap / Pagefile / Hibernation
- Document OS-level risk; optionally offer to disable hibernation / pagefile encryption guidance

### 3.35 JSON Deserialization of Untrusted Data
- If v3.10 imports JSON backups or configs, validate schemas before loading

### 3.36 AI / LLM Prompt Injection via Transaction Descriptions
- Relevant only if an LLM is introduced later; document as risk if categorization ever moves beyond traditional ML

### 3.37 Physical Device Auto-Lock / Screen Lock Integration
- Detect idle time and lock the app; integrate with OS screen-lock events

### 3.38 Secure Deletion of Unencrypted Temp Files
- Use secure overwrite or OS secure-delete APIs for sensitive temp files before unlinking

### 3.39 Printer / Cloud Print Warnings
- Same as 3.32; ensure any print flow warns against cloud printers
- Obfuscate or blank account numbers in sensitive UI views to reduce exposure to screen-capture malware
- Files: frontend components rendering account/transaction data
- CSP headers, secure `localStorage` handling, sanitization of imported data rendered in UI
- Files: `frontend/`, `backend/api.py` CORS/middleware

### 3.12 Secure First-Run Defaults
- Force strong master password at setup; reject weak passwords; disallow unencrypted startup
- Files: `backend/routers/auth.py`, first-run wizard

### 3.13 Import Path Attacks Beyond PDF
- OFX/QIF/CSV formula injection, macro payloads, crafted dates, malformed records
- Files: future import routers; current `phase3_pipeline/parsers/`

### 3.14 Denial-of-Service Limits
- Cap PDF file size, page count, total transactions per import, model file size
- Files: `backend/routers/upload.py`, `backend/parsers/generic_pdf.py`

### 3.15 Log Sanitization
- Ensure descriptions, account numbers, balances never appear in plain-text logs or error traces
- Files: all `print()` and logging call sites

### 3.16 Frontend Dependency Supply Chain
- npm audit/pinning, same rigor as Python deps; block install-time scripts
- Files: `frontend/package.json`

### 3.17 Key Recovery / Backup Passphrase
- Encrypted recovery key or passphrase-protected backup for master-password loss
- Files: `backend/local/backup.py`, `backend/local/crypto.py`

### 3.18 OCR / Image Pipeline Subprocess Sandboxing
- Poppler/pdf2image + Tesseract isolated with no network access
- Files: `backend/parsers/generic_pdf.py`, `backend/local/guards.py`

## 4. Out of Scope

The following are explicitly rejected for v3.10 because they violate local-first / private constraints:
- Snowflake, BigQuery, or any cloud data warehouse
- Plaid / Yodlee / live bank feeds
- Cloud OCR APIs
- Hosted analytics or telemetry services
- OAuth / Auth0 / external identity providers
- SaaS CI/CD with external secret storage (unless self-hosted)
