### TRACK R1 â€” P0 Critical: Double-Entry GL Auto-Posting
**Owner:** Jane  
**Branch:** `v3.11.6-dev-REMEDIATE-R1-gl-autopost` cut from `v3.11.6-dev`  
**Merge target:** `v3.11.6-dev`  
**ETA:** 3-4 days  

**Problem:** OFX/QFX, PDF/CSV upload, and manual transaction import create `Transaction` records but do **not** create `GeneralLedgerEntry` debit/credit pairs. The platform claims double-entry accounting but the GL is decoupled from imported transactions.

**Deliverables:**
1. `backend/accounting/gl_bridge.py` â€” new module with deterministic transaction-to-GL translation.
   - Map each transaction to a bank/cash asset account (debit for deposits, credit for withdrawals).
   - Map the offset to a default income/expense COA account based on a rules engine:
     - Institution + description keyword matching.
     - ML categorizer as fallback.
     - `Uncategorized Income` / `Uncategorized Expense` as final fallback.
   - Produce **balanced GL entries** for every transaction: `sum(debits) == sum(credits)`.
   - Support splits if a transaction is explicitly split by the user.
2. Integration points:
   - `POST /api/imports/ofx` â€” call GL bridge after statement transactions are persisted.
   - `POST /api/upload` (PDF/CSV) â€” call GL bridge after parser returns transactions.
   - `POST /api/transactions` (manual create) â€” call GL bridge.
   - `POST /api/recurring/materialize` â€” call GL bridge when recurring rules generate transactions.
3. Idempotency:
   - Use `txn_uid` to prevent duplicate GL entries if the same import is retried.
   - Store `import_source` and `source_id` on generated GL entries.
4. Router additions:
   - `POST /api/ledger/auto-post-batch` â€” optional admin endpoint to retroactively generate GL entries for transactions that lack them (idempotent).
5. Tests:
   - `backend/tests/test_gl_bridge.py` â€” unit tests for mapping logic, split handling, fallback accounts.
   - Extend `test_ofx.py`, `test_upload.py`, `test_recurring.py` to assert GL balance after import/materialization.
   - Meaningful assertion: `SELECT SUM(amount) WHERE type='debit' == SELECT SUM(amount) WHERE type='credit'` for generated entries.

**Acceptance criteria:**
- Importing a 10-row OFX statement produces exactly 20 GL rows (10 debit, 10 credit) and the trial balance balances.
- Manual transaction create produces balanced GL entries.
- Re-importing the same OFX is idempotent (no duplicate GL rows).
- SQLite + PostgreSQL regression suites pass.

**Dependencies:**
- COA seed must include `Uncategorized Income` (type: income) and `Uncategorized Expense` (type: expense). Verify seed; add if missing.
- ML categorizer must be callable without heavy model load in tests (mock or lightweight fallback).

### TRACK R2 â€” P0 Critical: Period Close Automation
**Owner:** Jane  
**Branch:** `v3.11.6-dev-REMEDIATE-R2-period-close`  
**ETA:** 2-3 days  

**Problem:** No `is_closed`, no closing entries, no Retained Earnings automation. Month/year-end close is manual.

**Deliverables:**
1. Schema changes (Alembic migration):
   - Add `is_closed`, `closed_at`, `closed_by_profile_id` to `Period` model.
   - Add `entry_type` enum column to `GeneralLedgerEntry` with values: `regular`, `adjusting`, `closing`.
2. Business logic in `backend/accounting/period_close.py`:
   - `close_period(client_id, period_id, profile_id)`:
     - Validate no prior unclosed earlier period for the same client (sequential close rule).
     - Compute net income = income total - expense total for the period.
     - Create closing GL entries:
       - Debit all income accounts by their period balance (zero them out).
       - Credit all expense accounts by their period balance (zero them out).
       - Credit Retained Earnings by net income if positive; debit if negative.
     - Mark all created entries `entry_type='closing'`.
     - Mark period `is_closed=True`.
   - `reopen_period()` â€” owner-only endpoint that reverses closing entries and clears the flag.
3. Hard guards:
   - Reject new transactions, journal entries, imports, and reconciliation actions for a closed period.
   - Reject edits to GL entries with `entry_type='closing'`.
4. Router additions:
   - `POST /api/periods/{id}/close`
   - `POST /api/periods/{id}/reopen`
5. Tests:
   - `backend/tests/test_period_close.py` â€” close a quarter, verify income/expense zeroed, retained earnings updated, reopen restores state, closed-period writes rejected.
   - Extend trial balance / P&L tests to verify closing entries do not corrupt period reports.

**Acceptance criteria:**
- Closing a period with $50K income and $30K expenses posts $20K net to Retained Earnings.
- Reopening the period restores the original income/expense balances and removes the closing entries.
- Attempting to import into a closed period returns 422.

**Dependencies:**
- Requires TRACK R1 (GL entries must exist for closing logic to have data to close).
- COA must include a `Retained Earnings` equity account or auto-create it on first close.

### TRACK R3 â€” P0 Critical: Reconciliation Locking
**Owner:** Jane  
**Branch:** `v3.11.6-dev-REMEDIATE-R3-reconciliation-lock`  
**ETA:** 2 days  

**Problem:** A completed reconciliation can still be modified; cleared transactions remain editable.

**Deliverables:**
1. Schema change:
   - Add `is_completed`, `completed_at`, `completed_by_profile_id` to `ReconciliationImport`.
2. Close reconciliation workflow:
   - `POST /api/reconciliation/{id}/complete` â€” sets `is_completed=True`, records cleared transaction IDs as a snapshot.
   - `POST /api/reconciliation/{id}/reopen` â€” owner/admin only, reopens for correction, clears snapshot.
3. Guards:
   - Completed reconciliation rejects `match`, `unmatch`, `update`, `delete` actions.
   - Transactions marked `cleared` by a completed reconciliation reject edits/voids/deletes unless the reconciliation is reopened.
   - If a transaction is part of a completed reconciliation, any direct transaction mutation returns 409 with message referencing the reconciliation ID.
4. Audit trail:
   - Log `reconciliation_completed` / `reconciliation_reopened` events with cleared transaction count and difference.
5. Tests:
   - `backend/tests/test_reconciliation_lock.py` â€” complete a reconciliation, assert subsequent mutations are rejected, assert reopen restores editability.

**Acceptance criteria:**
- After completing reconciliation, `POST /api/reconciliation/{id}/match` returns 409.
- After completing reconciliation, editing a cleared transaction returns 409.
- Reopening allows correction and re-completion.

**Dependencies:**
- None beyond existing reconciliation models.
