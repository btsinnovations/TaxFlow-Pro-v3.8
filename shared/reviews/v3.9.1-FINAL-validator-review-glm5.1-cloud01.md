## P0 — Token Revocation & Column Encryption

### P0.1: Application-Level Column Encryption ✅

**Files:** `backend/local/column_encryption.py`, `backend/local/crypto.py`, routers (`clients.py`, `accounts.py`, `upload.py`, `transactions.py`)

- AES-256-GCM with Argon2id key derivation, per-user salt stored in `users.encryption_salt`
- Runtime manager cache (thread-safe dict) — populated at boot/login, cleared at logout
- `encrypt_for_user()` / `decrypt_for_user()` with `_looks_encrypted()` heuristic for legacy plaintext fallback
- Encrypted columns: `clients.tax_id`, `accounts.account_number_masked`, `transactions.description`, `statements.filename`
- No-op fallback when manager is absent (app remains usable if encryption is disabled)
- **Risk noted:** `decrypt_for_user` falls back to returning ciphertext silently when no manager is cached. This is by design (graceful degradation) but means a codepath that reads encrypted data without logging in first will get ciphertext, not plaintext. The only codepath that reads data goes through `get_current_user` which caches the manager, so this is safe in practice.

### P0.2: Server-Side Token Revocation ✅

**Files:** `backend/auth.py`, `backend/routers/auth.py`, `backend/models.py`, migration `1116e8143fc6`

- JWT payload includes `jti` (32-byte URL-safe token) and `type: "access"`
- `RevokedToken` model with `jti` (unique indexed), `user_id`, `token_type`, `expires_at`, `revoked_at`
- `decode_access_token()` optionally accepts `db` session and rejects revoked tokens
- `/api/auth/logout` stores `jti` server-side AND clears column-encryption manager
- `_get_current_user()` passes db session to `decode_access_token()`, enforcing revocation on every protected route
- Idempotent revocation (revoke same token twice → returns existing record)
- **Observation:** No background cleanup of expired `RevokedToken` rows. Over months of usage, this table grows. The `expires_at` column exists but no cron/prune endpoint sweeps expired rows. **Non-blocking** for a single-tenant local app, but worth noting for long-running deployments.

### P0 Verdict: ✅ PASS

## P1 — OCR Parser, Parser Coverage, Frontend Cleanup

### P1.1: Parser Coverage Expansion ✅

**Files:** `backend/parsers/tdbank.py`, `backend/parsers/chime.py`, `backend/parsers/edfed.py`, `backend/parsers/queensborough.py`, `backend/parsers/institution.py`, `backend/parsers/parser_base.py`

- 4 new institution-specific parsers: TD Bank (checking + credit), Chime (checking + Credit Builder), EdFed (share draft + credit), Queensborough National Bank
- Shared `parser_base.py` provides `build_parse_result`, `extract_amounts`, `infer_year`, `make_tx`, `normalize_signed_amount`, `parse_date_us`
- Institution detection expanded with credit-card keywords (TD Bank credit, Chime Credit Builder/SpotMe, EdFed Rewards Visa)
- Whole-word matching for single-word markers in `institution.py` (`_marker_present` uses `\b` regex) — prevents false-positive "td" matching inside words
- `parse_statement_pdf()` routes to institution-specific parsers when detected, falls back to generic, returns `needs_review: true` for unknown institutions
- **Observation:** Queensborough `can_handle` matches `"qnb"` as a substring (no word-boundary). This could false-positive on text containing "qnb" inside longer words. Low risk for bank statements but technically inconsistent with the `\b` approach used in `_marker_present`. **Non-blocking.**

### P1.2: OCR Parser as First-Class Opt-In ✅

**Files:** `backend/parsers/ocr_parser.py`, `backend/parsers/institution.py`, `backend/routers/upload.py`, frontend

- `OCRPDFParser` class with configurable `language`, `dpi`, `grayscale`
- `supported()` class method checks for pdf2image, pytesseract, and Tesseract CLI availability
- Constructor raises clear `RuntimeError` with install instructions when OCR deps are missing
- Returns GenericPDFParser-compatible dict shape (`meta`, `reconciliation`, `transactions`, `pages`, `template`)
- `parse_statement_pdf()` routes to OCR when `force_ocr=True`, sets `needs_review: True`
- Upload router accepts `force_ocr` boolean form field, backward-compatible
- Frontend `UploadSection.tsx` has "Use OCR" switch
- `useAPI.ts` `uploadFile()` accepts optional `forceOcr`
- **Observation:** OCR parser does not perform transaction extraction itself — it returns raw text and empty transactions, deferring to the existing pipeline. This is documented and intentional.

### P1.3: Frontend Mock-Data Cleanup ✅

## P2 — Password Policy, PII Redaction, Backup/Restore, Migration Health, CI/CD, Version Bump

### P2.1: Password Policy (SEC-02) ✅

**Files:** `backend/utils/password_policy.py`, `backend/routers/auth.py`

- Minimum 12 characters
- Minimum 50 bits estimated entropy (Shannon + pool-size calculation, conservative)
- Rejects literal "password" substring
- Rejects username substring (when provided)
- Rejects common passwords (70+ entry blocklist including "taxflow" variants)
- Optional zxcvbn integration (score ≥ 3 required when installed)
- Enforced on `/auth/boot`, `/auth/register`, `/auth/change-password`
- Login flow untouched — existing users not locked out

### P2.2: PII Redaction (SEC-03) ✅

**Files:** `backend/utils/redaction.py`, `backend/audit/audit_trail.py`

- Regex-based redaction of: SSN (dashed + 9-digit), 16-digit card numbers, bank account numbers >6 digits, email addresses, US phone numbers
- `redact_pii()` for single strings, `redact_pii_in_json()` for recursive JSON structures
- Applied to `description` field and recursively to `details` JSON in every audit entry before DB insert
- JSON structure preserved; only string values transformed

### P2.3: Backup/Restore (DATA-01) ✅

**Files:** `backend/local/backup.py`, `scripts/backup.py`, `scripts/restore.py`

- `backup_db()` copies SQLite + SHA-256 manifest with timestamp and version
- `restore_db()` verifies SHA-256 hash before restoring, raises `BackupError` on mismatch
- `create_encrypted_backup()` / `restore_backup()` for AES-256-GCM encrypted archives
- `enable_wal()` and `integrity_check()` helpers for SQLite crash recovery
- CLI entry points for one-command backup/restore

### P2.4: Migration Health (DATA-02) ✅

**Files:** `backend/local/migration_health.py`, `backend/routers/health.py`, `backend/api.py`

- `check_migrations()` uses Alembic `ScriptDirectory` + `MigrationContext` to compare DB revision vs head
- Returns `{current, latest, pending, ok}` dict
- Startup logs warning if migrations pending; `STRICT_MIGRATIONS=true` causes `SystemExit(1)`
- `GET /api/health/migrations` (authenticated) and `GET /api/health/public` endpoints
- Health endpoints read version from `version.txt` (single source of truth = `3.9.1`)
