# Decision Record: v3.11.5 Tenant Boundary for RLS

## Status

**Approved and implemented** — applies to SQLite application-level scoping
immediately; PostgreSQL RLS policies deferred to RLS.03/RLS.04/RLS.06 pending
orchestrator go-ahead.

## Decision

**Tenant = Client.**

In TaxFlow Pro v3.11.5, the tenant boundary is the `Client` row. Every
multi-tenant entity (accounts, statements, transactions, rules, flags, GL
entries, audit entries, etc.) carries a non-nullable `tenant_id` foreign key
to `clients.id`.

### Rationale

- The application already has a `Client` model that users create and select.
- `Account`, `Statement`, and `Transaction` already include `tenant_id`.
- Single-user mode has exactly one user with one or more clients; the active
  tenant resolves to the user's primary client.
- Multi-entity mode uses `X-Tenant-ID` to pick a client.
- This avoids inventing a separate "tenant" table and migrating existing data.

## Consequences

- SQLite fallback cannot use DB-level RLS, so all router queries must filter by
  `tenant_id` (and `user_id`) at the application layer.
- PostgreSQL policies will reference `tenant_id` columns when RLS.03/RLS.04
  are approved.
- No schema change is required for v3.11.5 beyond existing migrations.
